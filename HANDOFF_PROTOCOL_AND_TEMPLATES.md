# Hand-off protocol + Starter template files

## Agent hand-off protocol

When one agent finishes and passes to the next.

### Hand-off format

```
FROM: {agent-name} ({stage})
TO: {next-agent} ({next-stage})
STATUS: {ready | blocked | has-risks}
ARTIFACT: {what was produced}
MEMORY_UPDATES: {files changed in .ai/memory/}
```

### Example: Planner → Architect

**Planner finishes plan**:

```markdown
FROM: Claude (Planner)
TO: Gemini (Architect)
STATUS: ready
ARTIFACT: plan.md (in active-tasks.md#FEAT-042)
MEMORY_UPDATES:
- active-tasks.md (FEAT-042 planner stage ✓)
- decisions.md (added: "Use idempotency keys for retry safety")

## What Architect should know

**Plan summary**: Exponential backoff retry handler for payments
**Files to touch**: 5 (retry-handler, state-machine, migration, tests, config)
**Key constraint**: Kafka must use exactly-once semantics
**Key risk**: Retry storm if circuit breaker fails (mitigation: max 5 retries)
**Dependencies**: None (self-contained feature)
**Blocks**: FEAT-043 (invoice generation depends on this)

## What Architect should design

1. State machine for retry lifecycle
2. Idempotency key strategy (UUID generation, storage)
3. Circuit breaker pattern (Hystrix + config)
4. Kafka topic strategy (payment-failed, payment-retrying, payment-succeeded)

## When design is complete

Update active-tasks.md#FEAT-042:
- Mark architect stage ✓
- Add design doc link
- Update "Next agent → Impl"
```

**Architect reads this, does design, hand off to Impl**:

```markdown
FROM: Gemini (Architect)
TO: Claude (Implementation)
STATUS: ready
ARTIFACT: design-doc.md (in active-tasks.md#FEAT-042)
MEMORY_UPDATES:
- active-tasks.md (FEAT-042 architect stage ✓)
- architecture.md (added Retry Service to Payment domain)

## Design summary

State machine: PENDING → RETRYING → SUCCESS | FAILED
Idempotency: UUID generated by caller, stored in payments.idempotency_key
Database: Add 2 columns (idempotency_key UUID unique, retry_count INT)
Circuit breaker: Hystrix 5s timeout, 50% failure threshold
Kafka: Subscribe to payment-failed, emit payment-retrying/succeeded

## What Impl should code

1. RetryHandler service (state machine logic)
2. Circuit breaker integration (Hystrix bean config)
3. Database migration (alter table payments)
4. Test suite (state transitions, idempotency, circuit breaker)
5. Kafka integration (consume failed events, emit retrying)

## Code patterns to follow

See memory/coding-style.md for:
- Dependency injection pattern (@Autowired on field)
- Error handling (@ControllerAdvice)
- Testing (Testcontainers, not H2)

## Risks discovered during design

- Race condition on idempotency_key unique constraint
  → Mitigation: optimistic locking on Payment entity
- Circuit breaker threshold too sensitive (50% failures)
  → Decision: log every circuit breaker trip to monitoring

## When impl is complete

Update active-tasks.md#FEAT-042:
- Mark impl stage ✓
- Link to PR/commit
- Update "Next agent → QA"
```

**Impl hands off to QA**:

```markdown

## Contract compliance

✓ Summary included

2. Coverage meets requirement (92% > 90% for payment module)
3. State machine paths covered:
   - Normal retry → success
   - Exhausted retries → failed

5. Migration safe (can rollback without data loss)

### .ai/rules/global/security.md

```markdown
# Global security rules

## API authentication

NEVER:
- Hardcode API keys in code
- Log authentication tokens
- Send secrets in query params (use headers)
- Skip TLS (always HTTPS)

ALWAYS:
- Use environment variables or secrets vault (Sealed Secrets, AWS Secrets Manager)
- Validate JWT token signature
- Set short expiration (15min access, 7d refresh)
- Implement rate limiting (100 req/min per IP default)

## Database security

NEVER:
- SQL injection (always parameterized queries)
- Log password fields
- Store plaintext passwords (use bcrypt/Argon2)

ALWAYS:
- Encrypt at rest (if handling PII/payment data)
- Audit trail for data modifications (created_at, updated_at, updated_by)
- Restrict data access by role (RBAC)

## Secret management

All secrets stored in:
- Development: .env (git-ignored)
- CI/CD: GitHub Secrets or CI provider
- Production: Sealed Secrets (K8s) or managed service (AWS Secrets Manager)

Pattern:
```
export DB_PASSWORD=$(kubectl get secret payment-service -o jsonpath='{.data.password}' | base64 -d)
```

Never:
- Commit .env
- Log secrets in stack traces
- Hardcode in Dockerfile/docker-compose

## Code review checklist

Before merging any PR:
- [ ] No credentials in code
- [ ] No SQL injection vectors
- [ ] TLS enforced
- [ ] Rate limiting configured
- [ ] Error messages don't leak system details
- [ ] Logging excludes sensitive fields
```

# Java conventions (Spring Boot)

## Project structure

```
services/
├── payment-service/
│   ├── src/main/java/com/company/payment/
│   │   ├── controller/    (REST endpoints)
│   │   ├── service/       (business logic)

│   │   ├── dto/           (data transfer objects)
│   │   ├── exception/     (custom exceptions)
│   │   ├── config/        (beans, properties)
│   │   └── PaymentServiceApplication.java
│   ├── src/test/java/... (mirrors main, +Test suffix)
│   ├── pom.xml
│   └── application.yml (externalized config)
```

## Dependency injection

ALWAYS use field injection (current convention):

```java
@Service
public class PaymentService {
  @Autowired
  private PaymentRepository repo;
  
  @Autowired
  private KafkaTemplate<String, PaymentEvent> kafka;
}
```

Being phased out: Constructor injection (legacy — don't add new code)

## Error handling

Pattern: Custom exception + @ControllerAdvice

```java
// exception/PaymentNotFound.java
public class PaymentNotFound extends RuntimeException {
  public PaymentNotFound(UUID id) {
    super("Payment " + id + " not found");
  }
}

// controller/ExceptionHandler.java
@ControllerAdvice
public class GlobalExceptionHandler {
  @ExceptionHandler(PaymentNotFound.class)
  public ResponseEntity<?> handle(PaymentNotFound e) {
    return ResponseEntity.notFound().build();
  }
}
```

## Testing

Use: JUnit 5 + Mockito + Testcontainers (not H2)

```java
@SpringBootTest
class PaymentServiceTest {
  @Mock
  private PaymentRepository repo;
  
  @InjectMocks
  private PaymentService service;
  
  @Test
  void testRetrySuccess() {
    Payment p = new Payment(...);
    when(repo.save(any())).thenReturn(p);
    
    Payment result = service.retry(p);
    assertThat(result.getRetryCount()).isEqualTo(1);
  }
}
```

DO NOT use H2 in-memory for integration tests. Use real DB (Testcontainers + PostgreSQL).

## Naming

- Classes: PascalCase (PaymentService, PaymentRetryHandler)
- Methods: camelCase (retryPayment, getPaymentById)
- Constants: UPPER_SNAKE_CASE (MAX_RETRIES, DEFAULT_TIMEOUT_MS)
- Packages: lowercase.dot.separated (com.company.payment.service)

### .ai/rules/backend/sql.md

```markdown
# SQL conventions

## Migrations

Tool: Flyway or Liquibase

File naming: `V{version}__description.sql`
Example: `V002__add_retry_state.sql`

```sql
-- V002__add_retry_state.sql

ALTER TABLE payments ADD COLUMN idempotency_key UUID UNIQUE;
ALTER TABLE payments ADD COLUMN retry_count INT DEFAULT 0;
ALTER TABLE payments ADD COLUMN last_retry_at TIMESTAMP;

-- backfill: generate UUID for existing records (safe default)
UPDATE payments SET idempotency_key = gen_random_uuid() 
  WHERE idempotency_key IS NULL;

ALTER TABLE payments ALTER COLUMN idempotency_key SET NOT NULL;
```

## Query safety

NEVER: String concatenation for queries

```java
// WRONG
String query = "SELECT * FROM payments WHERE id = " + id;
repo.query(query);

// RIGHT
@Query("SELECT p FROM Payment p WHERE p.id = :id")
Payment findById(@Param("id") UUID id);
```

## Performance

Index high-cardinality columns used in WHERE:

```sql
CREATE INDEX idx_payments_user_id ON payments(user_id);
CREATE INDEX idx_payments_status ON payments(status);
CREATE INDEX idx_payments_created_at ON payments(created_at DESC);
```

Avoid: Full table scans, N+1 queries, missing LIMIT

```java
// WRONG: N+1 query
List<Payment> payments = repo.findAll();
for (Payment p : payments) {
  User u = userRepo.findById(p.getUserId()); // N queries!
}

// RIGHT: JOIN
@Query("SELECT p FROM Payment p LEFT JOIN FETCH p.user WHERE p.status = :status")
List<Payment> findByStatusWithUser(@Param("status") String status);
```

## Naming

- Tables: singular, lowercase, snake_case (payment, order, order_item)
- Columns: lowercase, snake_case (user_id, created_at, is_active)
- Primary key: id (UUID default for distributed systems)
- Foreign keys: {table}_id (user_id references users(id))
- Timestamps: created_at, updated_at (both DEFAULT CURRENT_TIMESTAMP)
```

### .ai/memory/decisions.md (skeleton)

```markdown
# Architecture Decision Records (ADRs)

## ADR-001: UUID for distributed IDs

**Status**: Accepted
**Date**: 2025-04-01
**Owner**: @architect-lead

### Decision
Use UUID v4 for all primary keys instead of auto-increment integers.

### Reason
- Distributed system compatibility (multiple writers)
- No coordination needed to generate IDs
- Privacy (ID doesn't leak sequence)

### Consequence
- Larger index size (16 bytes vs 8 bytes)
- Slower generation (crypto randomness)
- Mitigation: Pre-generate UUIDs on app side, not DB side

### Alternatives considered
- Snowflake IDs (database coupling)
- Sequential UUIDs (UUID v6, new but less tested)

---

## ADR-002: Event-driven architecture for payment processing

**Status**: Accepted
**Date**: 2025-04-02

### Decision
Use Kafka for payment events (payment-created, payment-succeeded, payment-failed).

### Reason
- Decoupling: payment service doesn't know about refund service
- Auditability: immutable event log
- Scalability: easy to add new consumers later

### Consequence
- Eventual consistency (refund may be delayed 100ms)
- Needs idempotency (same event processed twice → same result)

---

## ADR-003: Single source of truth pattern (no caching)

**Status**: Accepted
**Date**: 2025-04-05

### Decision
No client-side or server-side caching for critical business data (orders, payments).

### Reason
- Reduce cache invalidation complexity
- PCI-DSS compliance (payments always verified fresh)
- Simplify audit trail

### Consequence
- Higher database load
- Mitigation: Read replicas + indexed queries + pagination

---

## ADR-004: Idempotency via UUID deduplication

**Status**: Accepted
**Date**: 2025-05-09

### Decision
Idempotency implemented via UUID: caller generates UUID, we store it + check for duplicates.

### Reason
- No need for distributed transaction coordinator
- Clients already know what they're doing
- Works across retries

### Example
```
POST /payments { idempotency_key: "uuid-1", amount: 100 }
Response: { id: "pay-123", status: "success" }

Same request again:
POST /payments { idempotency_key: "uuid-1", amount: 100 }
Response: { id: "pay-123", status: "success" } ← same result
```

---

## ADR-005: Exponential backoff for payment retries

**Status**: Proposed (not yet accepted)
**Date**: 2025-05-09

### Decision
Implement automatic retry with exponential backoff (1s → 2s → 4s → 8s → 16s, max 5 attempts).

### Reason
- Transient failures recover without manual intervention
- Exponential spacing reduces thundering herd
- 5 attempts over 32s covers most transient issues

### Consequence
- May process same payment twice (idempotency required)
- Circuit breaker needed (if >50% failures, skip retries)

### Alternative: Fixed backoff (1s between all retries)
- Simpler to understand
- Higher risk of thundering herd during outages

---

## How to use these templates

1. **Create .ai/ directory** in your repo root
2. **Copy structure** (rules/, memory/, contracts/, workflows/, agents/)
3. **Populate with your data**:
   - rules/: Add domain-specific rules (payment, frontend, ibm-bpm)
   - memory/: Write your architecture.md, decisions.md, coding-style.md
   - contracts/: Define your output requirements
   - agents/: Customize for your team
4. **Create AGENTS.md** in root (entry point) — see above
5. **Start tracking active-tasks.md** as work begins
